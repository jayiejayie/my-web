# 如何开展安全风险分级管控（V2 精炼版）

> 基于蝴蝶结模型（Bow-Tie）的工程实践分享
>
> 以基坑土方开挖为案例，把七步流程讲透

---

## 写在前面：一个问题，三个痛点

2021年新《安全生产法》将双重预防机制写入法律，风险分级管控已经从"可选做法"变成了法定责任。

但回到工程现场，一个现实的问题摆在每个项目面前：**这到底怎么干？**

很多项目的做法是让安全员加班赶出一套体系文件，填几张评价表，挂几块公示牌。至于这些表格和现场实际施工有多大关系、风险辨识有没有遗漏、管控措施能不能落地——往往没有经过认真推敲。

传统做法有三个通病：

| 痛点 | 表现 | 后果 |
|------|------|------|
| **辨识靠经验，盲区多** | 安全员凭个人经验或照搬其他项目清单 | 没人能保证"我们有没有漏掉什么" |
| **措施太笼统，缺层级** | "加强管理""严格执行规程" | 无法指导现场行动，高风险和低风险的措施没区别 |
| **清单和排查两张皮** | 管控清单锁在柜里，排查用另一套表 | 双控变成"应付检查"的表面文章 |

这三个痛点的症结是同一个：**缺少一个能把风险"可视化、结构化、可验证"的分析工具。**

本文要介绍的，就是这个工具——**蝴蝶结模型（Bow-Tie）**。用一个贯穿全文的案例——**基坑土方开挖**——从头到尾走一遍风险分级管控的实施流程。

---

## 第1章 两句话讲透核心框架

### 1.1 一条因果链

事故不是孤立的事件，而是一条失效链条的终点：

```
危险源 → 威胁 → 管控措施失效 → 初始事件（失控）→ 应急措施失效 → 后果
```

| 环节 | 通俗理解 | 土方案例 |
|------|---------|---------|
| **危险源** | 能量的载体（客观存在） | 基坑边坡土体（重力势能） |
| **威胁** | 直接触发失控的因素 | 坡度过陡、降雨侵蚀 |
| **初始事件** | 危险源失控的中性状态 | 基坑边坡坍塌 |
| **后果** | 能量释放造成的损失 | 人员被埋伤亡 |

越早干预，成本越低，效果越好。

### 1.2 两道防线

```
第一道防线：风险分级管控 —— 干预"危险源 → 初始事件"
   核心任务：辨识危险源、评估风险、设计控制措施

第二道防线：隐患排查治理 —— 干预"初始事件 → 后果"
   核心任务：排查隐患、整改闭合、防止失控演变成事故
```

本文聚焦第一道防线。一句话：**把风险控制在隐患形成之前。**

### 1.3 什么是蝴蝶结（Bow-Tie）模型？

Bow-Tie 以**初始事件**为中心，左右对称展开：

```
  威胁1 ──→ [控制措施] ──┐           ┌── [应急措施] ──→ 后果1
                       │           │
  威胁2 ──→ [控制措施] ──┼→ 初始事件 ──┼── [应急措施] ──→ 后果2
                       │           │
  威胁3 ──→ [控制措施] ──┘           └── [应急措施] ──→ 后果3
```

它和传统方法的本质区别：

| | 传统方法 | Bow-Tie方法 |
|--|---------|------------|
| **思维** | 单线："危险源→措施" | 链式："危险源→威胁→措施→事件→应急→后果" |
| **针对性** | 笼统描述 | 每个威胁对应专门措施 |
| **可验证** | 措施写了，但不知道有没有用 | 每条措施对应一个保障活动 |
| **与排查联动** | 两张皮 | 保障活动直接变成排查项 |

## 第3章 七步实施流程

### 3.1 第一步：谁来干？按什么规矩干？

在开始任何技术分析之前，先解决组织和制度问题。

**组织架构**：企业层面由主要负责人任组长，项目层面由项目经理任组长。关键一条：**班组长必须纳入工作小组**——他们是一线落实的关键角色。

**五项基本制度**：风险分级管控制度、隐患排查治理制度、教育培训制度、考核奖惩制度、文件管理制度。不需要写得多复杂，但必须说清楚"谁在什么时候做什么事"。

**分层培训**：不同层级学不同的内容——

| 层次 | 对象 | 重点 |
|------|------|------|
| 决策层 | 企业领导、项目班子成员 | 政策要求、体系建设思路 |
| 管理层 | 技术/安全管理人员 | 评估方法、清单编制 |
| 执行层 | 班组长、分包负责人 | 隐患排查、措施落实 |
| 操作层 | 一线作业人员 | 本岗位风险、应急响应 |

**案例实操**：土方项目开工前，项目经理组织成立双控工作小组（安全总监+技术负责人+施工员+机械管理员+班组长）。编制了实施方案和培训计划，开工前组织了三场分层培训。

---

### 3.2 第二步：风险点在哪？（划范围）

**什么是风险点**："可能出事的地方或活动"，分作业活动类和设备设施/部位区域类。

**怎么找？用三维扫描法**：

| 扫描维度 | 扫描什么 |
|---------|---------|
| 部位/区域 | 具有潜在风险的物理位置或空间 |
| 设施/设备 | 可能引发风险的设备、设施或工具 |
| 作业活动 | 伴随风险的作业过程 |

**案例实操**：土方工程三维扫描结果：

```
土方工程风险点（5个）
  │
  ├── 作业活动扫描
  │     ├── RP-001 基坑土方机械开挖作业
  │     └── RP-003 人工清底修坡作业
  │
  ├── 设备设施扫描
  │     ├── RP-002 挖掘机作业
  │     └── RP-004 土方运输车辆作业
  │
  └── 部位区域扫描
        └── RP-005 基坑边土方堆放区
```

验证方法：列出常见事故类型（坍塌、车辆伤害、机械伤害、高处坠落），检查每个类型是否可以追溯到某个风险点。验证通过。

---

### 3.3 第三步：危险源是什么？（找根源）

危险源是**承载能量的物体**，是风险的"根"。

**能量源扫描法（四问法）**：

| 问题 | 扫描目标 |
|------|---------|
| 什么有质量？ | 物体、材料、土体、设备 |
| 什么有势能或动能？ | 高处物体、移动机械 |
| 什么有压力？ | 侧向土压力、水压力 |
| 什么有化学能？ | 危险化学品、燃气 |

**案例实操**：土方工程识别出4个危险源：

```
危险源扫描结果
  │
  ├── H001 基坑边坡土体（势能）—— 土体具有重力势能，边坡失稳时释放
  ├── H002 挖掘机（动能）—— 机械运动部件具有动能
  ├── H003 运输车辆（动能）—— 车辆行驶具有动能
  └── H004 堆放土方（势能）—— 堆土具有重力势能
```

注意一个关键区分：**"坡度过陡""违章操作"不是危险源，它们是后续要分析的"威胁"。** 危险源只有这几样承载能量的"物体"。

---

### 3.4 第四步：风险到底有多大？（定级别）

这一步是整个分析的核心。我们需要依次识别出：**初始事件→威胁→后果→风险等级**。

#### 初始事件——风险分析的"分水岭"

初始事件是危险源**失控的中性状态**，不是灾难后果。

三条判断原则：
- **中性描述**：描述"失控状态"而非"灾难后果"
- **单一事件**：聚焦一个明确的失控节点
- **可识别性**：能通过工程或管理措施进行控制

检验方法："人员被埋"不是初始事件（这是后果），"基坑边坡坍塌"才是。

**案例实操**：对4个危险源，识别出8个初始事件：

```
H001 基坑边坡土体 → IE001 基坑边坡坍塌 / IE005 边坡局部坍塌
H002 挖掘机       → IE002 机械碰撞碾压 / IE003 机械失控 / IE004 机械倾覆
H003 运输车辆     → IE006 车辆碰撞碾压 / IE007 车辆失控
H004 堆放土方     → IE008 堆土滑塌
```

#### 威胁——什么会触发失控？

从"人-物-环"三个维度展开：

| 维度 | 扫描对象 | 土方案例（针对H001） |
|------|---------|-------------------|
| 人（行为） | 不安全行为 | T006 超挖 |
| 物（状态） | 设备、材料异常 | T001 坡度过陡、T002 堆载超限、T012 支护失效 |
| 环（环境） | 环境条件突变 | T003 降雨侵蚀、T004 机械振动、T005 不良地质 |

> **为什么不列"管理因素"？** 管理不是威胁，而是控制措施的载体。威胁是直接导致失控的因素——坡度过陡、降雨侵蚀——这些都是可以直接观察到的现象。"管理制度不健全"不会直接触发事故，它是通过"坡度过陡没人管"才间接发挥作用的。

**闭包验证**来检验威胁辨识是否完整：参照GB 6441，土方工程事故类型有坍塌、车辆致害、机械致害、高处坠落。检查所有威胁是否覆盖这些类型，确保无遗漏。

#### 后果——失控后最坏的结果

**案例实操**：以H001为例——

```
IE001 基坑边坡坍塌 → C001 人员被埋伤亡 / C002 设备损毁 / C005 工期延误
IE002 机械碰撞碾压 → C003 人员伤亡
IE004 机械倾覆     → C004 设备损毁
```

#### LS风险评估——定等级

采用LS法（R = L × S）：

| 风险等级 | 颜色 | R值 | 管控层级 |
|---------|:----:|:---:|---------|
| 高风险 | 红 | 15～25 | 企业级 |
| 中风险 | 橙 | 10～12 | 项目部级 |
| 低风险 | 黄 | 5～8 | 班组级 |
| 很低风险 | 蓝 | 1～4 | 岗位级 |

**案例实操**：

| 后果 | 危险源 | L | S | R | 等级 |
|------|-------|:-:|:-:|:-:|:----:|
| 人员被埋伤亡 | H001 | 3 | 5 | **15** | **高风险（红）** |
| 人员伤亡 | H002 | 3 | 5 | **15** | **高风险（红）** |
| 设备损毁 | H002 | 3 | 4 | **12** | **中风险（橙）** |

### 3.5 第五步：怎么管？（定措施）

措施分两类：**控制措施**（阻止威胁触发初始事件——左侧屏障）和**应急措施**（减轻后果损失——右侧屏障）。

#### 控制措施——五层级纵深防御

| 优先级 | 层级 | 编码 | 示例 |
|:------:|:-----|:----:|:-----|
| 1 | 消除 | E | 取消高危作业、改变工艺 |
| 2 | 替代 | S | 机械替代人工、预制替代现浇 |
| 3 | 工程控制 | EC | 物理隔离、限位装置、监测系统 |
| 4 | 管理控制 | MC | 操作规程、审批程序、技术交底 |
| 5 | 个体防护 | PPE | 安全帽、安全带 |

**纵深防御规则**：高风险路径必须有EC + MC 两层；中风险必须有MC。

**案例实操**：以H001（基坑边坡土体）为例，为每个威胁设计控制措施：

```
H001 基坑边坡土体
  │
  ├── T001 坡度过陡 → CM001 按1:0.5放坡(EC) + CM002 坡度复核验收(MC)
  ├── T002 堆载超限 → CM003 堆土距离≥3m限载(MC) + CM004 限位标识线(EC)
  ├── T003 降雨侵蚀 → CM005 排水沟+集水坑(EC) + CM006 暴雨停止坑边作业(MC)
  ├── T004 机械振动 → CM007 挖掘机距坑边≥1m限位(EC)
  ├── T005 不良地质 → CM008 边坡位移监测及预警(EC)
  ├── T006 超挖     → CM009 基底标高复核验收(MC)
  └── T012 支护失效 → CM010 支护方案专家论证(EC) + CM011 施工过程质量验收(MC)
```

8个威胁共设计11道控制措施。按纵深防御规则检查：每个威胁路径上都设置了工程控制（EC）或管理控制（MC），高风险路径同时设置了两类措施，满足要求。

#### 应急措施——事发后减损

| 层级 | 响应时机 | 示例 |
|:----:|:---------|:-----|
| 工程控制 | 事发时自动/即时响应 | 逃生通道、急救箱、消防器材 |
| 管理控制 | 事发后人工响应 | 应急预案、疏散路线 |
| 个人防护 | 减轻伤害程度 | 安全帽、安全带 |

**案例实操**：以基坑边坡坍塌（IE001）为例

```
IE001 基坑边坡坍塌
  │
  ├── EM001 沿基坑每20m设逃生通道（EC）
  ├── EM002 配备应急救援物资（EC）
  ├── EM003 编制坍塌应急预案并定期演练（MC）
  └── EM004 与周边医疗机构建立应急联动（MC）
  │
  ▼
C001 人员被埋伤亡
```

#### 从分析成果到管控清单

把前面所有成果汇总为一份可执行的清单，每条记录包含：
- **场景信息**：分部工程、分项工程、工序
- **风险信息**：危险源→威胁→初始事件→后果
- **评估信息**：L、S、R、风险等级
- **措施信息**：控制措施、应急措施
- **责任信息**：管控层级、责任人

---

### 3.6 第六步：高/中风险措施评估

对高风险和中风险，在措施实施前必须做专项评估：

- **可行性**：技术、经济、工期上是否可行？
- **安全性**：实施后有没有次生风险？
- **可靠性**：能否持续有效，不会因日常磨损而失效？

---

### 3.7 第七步：告知、分级、验证、联动

#### 风险告知——让每个人都知道

| 方式 | 适用对象 | 核心要求 |
|------|---------|---------|
| 风险公示牌 | 高、中风险 | 显著位置公示风险点、等级、措施、责任人 |
| 岗位告知卡 | 每个工种/岗位 | 标明主要风险、管控措施、应急措施 |
| 安全技术交底 | 作业前人员 | 详细告知风险和措施 |
| 班前会 | 每日作业人员 | 讲解当日风险点和防范措施 |

#### 分级管控——谁的风险谁管

| 风险等级 | 管控层级 | 土方案例 |
|:--------:|:---------|:---------|
| 高风险（红） | 企业级 | 基坑边坡坍塌 → 企业挂牌督办 |
| 中风险（橙） | 项目部级 | 机械倾覆 → 项目经理主责 |
| 低风险（黄） | 班组级 | 人工清底 → 班组长落实 |
| 很低风险（蓝） | 岗位级 | 常规作业 → 员工自查 |

注意：上级管控的风险，下级同样要落实——不是"分而不管"，而是明确主责。

#### 关键保障活动——怎么知道措施还"好使"？

措施制定了，但**怎么知道它真的在起作用？**

每个控制措施和应急措施，都必须有对应的**保障活动**——验证措施是否持续有效。

| 类别 | 验证问题 | 案例 |
|:----:|:---------|:-----|
| A.检查维修 | 装好了没？还好着吗？ | 放坡坡度检查、排水沟是否通畅 |
| B.教育培训 | 用的人会吗？ | 应急预案演练 |
| C.验证测试 | 设计对吗？功能靠谱吗？ | 基坑支护方案专家论证 |
| D.独立审计 | 自检结果真实吗？ | 公司级专项检查 |

没有保障活动的措施，理论上无法验证其有效性。

#### 从管控清单到隐患排查清单——关键的"最后一公里"

**保障活动就是隐患排查的来源。**

```
管控清单中的一条措施：
  CM003 严格限制堆土距离≥3m（MC）
  ↓ 对应的保障活动：
  CA002 检查堆土距坑边距离是否≥3m（每班次）
  ↓ 纳入隐患排查清单：
  排查项：堆土距基坑边的距离是否≥3m
  排查方法：现场测量
  排查频次：每班次
```

这个转化逻辑确保了"两张清单"不是各自独立的——隐患排查清单的每一条，都对应着风控清单中某条措施的保障活动。反过来，排查出的每一个"不合格"，都可以追溯到某条管控措施失效。

#### 完整因果链全景图（以H001为例）

```

【预防侧】
```

## 第4章 让体系活下去（持续改进）

### 4.1 PDCA循环

```
P（策划）→ D（实施）→ C（检查）→ A（改进）
```

变化是常态，不变才是例外。持续改进的本质，就是用"有意识的管理节奏"来应对"不可预测的外部变化"。

### 4.2 动态更新触发条件

以下情况必须更新清单：
- 法规标准变化 → 周边环境变化 → 工艺/措施变化
- 工程进度推进 → 发生事故或险情后

**案例**：基坑开挖至基底前，深度变化导致边坡稳定性风险变化 → 重新评估 → 更新清单 → 进入新管控周期。

### 4.3 各方职责一句话

| 责任方 | 一句话职责 |
|:-------|:-----------|
| **建设单位** | 开工前牵头风险识别，保障经费，督促各方落实 |
| **施工单位** | 建立体系，编清单，执行管控（主体责任） |
| **监理单位** | 审查资料，巡查现场，责令停工（监理责任） |
| **勘察设计单位** | 注明安全风险重点部位，列明危大工程清单 |

## 结语：三个问题检验真落地

回到最开头那个问题：**风险分级管控到底怎么干？**

三步：**搭好架子、想透风险、持续改进。**

一个项目做得好不好，不看体系文件有多厚，而看这三个问题：

1. **风险清单和施工现场的真实风险对得上吗？**
2. **隐患排查清单是从风险管控清单转化过来的吗？**
3. **班组长和作业人员知道自己岗位的风险是什么吗？**

三个问题都答"是"，双控就是真的在落地。

> **"把风险控制在隐患形成之前，把隐患消灭在事故前面。"**
>
> ——国务院安委办〔2016〕11号

### Bow-Tie方法的五个核心价值

| 维度 | 传统局限 | Bow-Tie提升 |
|:-----|:---------|:------------|
| **可视化** | 清单表格，难直观理解 | 因果树状图，一目了然 |
| **结构化** | 靠经验，容易遗漏 | 威胁闭包验证，系统化覆盖 |
| **措施设计** | 笼统描述 | 五层级纵深防御，高风险必有工程控制 |
| **可验证** | 写了不知道有没有用 | 每条措施有对应的保障活动 |
| **与排查联动** | 两张皮 | 保障活动直接映射为排查项 |

这套方法不是用来取代双控的，而是让双控的每个环节做得更扎实。它的最大价值是：**让风险管理从"交作业"变成"真在用"。**
